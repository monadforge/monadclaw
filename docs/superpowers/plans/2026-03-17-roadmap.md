## Phase 1 — Session Persistence + System Prompt

**Goal:** conversations survive server restarts; agent has a configurable identity.

### Config changes

```toml
# config.toml
system_prompt = "You are a helpful personal assistant."
```

### New crate: `crates/store`

```
crates/store/src/
├── lib.rs
├── db.rs          # sqlx pool setup, migrations
└── session.rs     # Session CRUD, message history
```

**Schema:**

```sql
CREATE TABLE sessions (
    id          TEXT PRIMARY KEY,
    created_at  INTEGER NOT NULL,
    updated_at  INTEGER NOT NULL
);

CREATE TABLE messages (
    id          TEXT PRIMARY KEY,
    session_id  TEXT NOT NULL REFERENCES sessions(id),
    role        TEXT NOT NULL,   -- 'user' | 'assistant' | 'system'
    content     TEXT NOT NULL,
    created_at  INTEGER NOT NULL
);
```

### API changes

| Endpoint | Change |
|----------|--------|
| `POST /api/v1/sessions` | create session → return `{ id }` |
| `GET /api/v1/sessions/:id` | get session metadata |
| `GET /api/v1/sessions/:id/messages` | message history |
| `POST /api/v1/sessions/:id/chat` | send message (replaces stateless `/chat`) |

The old `POST /api/v1/chat` (stateless, frontend sends full history) stays as a
compatibility fallback.

### AppState changes

```rust
pub struct AppState {
    pub config:    Arc<Config>,
    pub provider:  Arc<dyn Provider>,
    pub store:     Arc<SessionStore>,    // new
    pub dashboard_password: Option<Arc<String>>,
}
```

### Dashboard changes

- Sidebar lists sessions (fetch `/api/v1/sessions`)
- Chat page sends to `/api/v1/sessions/:id/chat` instead of `/api/v1/chat`
- Session picker / "New chat" button

**Delivers:** persistent memory across restarts, configurable personality.

## Phase 2 — Tool Execution

**Goal:** agent can call tools (bash, file read/write) inside a turn.

### New crate: `crates/tools`

```
crates/tools/src/
├── lib.rs
├── tool.rs        # Tool trait + ToolCall + ToolResult
├── registry.rs    # ToolRegistry — register + dispatch
└── builtin/
    ├── bash.rs    # execute shell command, return stdout/stderr
    ├── read.rs    # read file content
    └── write.rs   # write file content
```

**Tool trait:**

```rust
pub trait Tool: Send + Sync {
    fn name(&self) -> &str;
    fn description(&self) -> &str;
    fn schema(&self) -> serde_json::Value;    // JSON Schema for LLM
    async fn call(&self, args: serde_json::Value) -> ToolResult;
}

pub struct ToolResult {
    pub output: String,
    pub is_error: bool,
}
```

### Agent loop changes (`crates/agent`)

`AgentSession::advance()` becomes a full agentic loop:

```
loop:
  1. call provider with tools attached
  2. if response has text only → done
  3. if response has tool_calls:
     a. dispatch each call to ToolRegistry
     b. append tool results as messages
     c. loop
  until: text response OR max_iterations reached
```

**Delivers:** agent can browse files, run commands, act on the system.

## Phase 3 — Discord Bot

**Goal:** talk to the agent through Discord DMs and channels.

### New crate: `crates/discord`

```
crates/discord/src/
├── lib.rs
├── bot.rs         # serenity event handler
└── handler.rs     # on_message → create/resume session → stream reply
```

**Flow:**

1. DM or `@mention` arrives
2. Look up session by `(guild_id, channel_id, user_id)` in store
3. Create if missing; call `run_turn(session, message, provider)`
4. Stream reply back (Discord has 2000 char limit → split or edit messages)
5. Commit exchange to store

### Config:

```toml
[discord]
bot_token_env = "DISCORD_BOT_TOKEN"
allowed_user_ids = ["123456789"]   # allowlist — empty = DMs closed
```

### Binary:

- Either add to `apps/server/main.rs` as an optional tokio task
- Or separate `apps/bot/` binary

**Delivers:** chat with the agent from any device via Discord.

## Phase 4 — Long-term Memory

**Goal:** agent remembers facts across sessions (user preferences, past events).

### New crate: `crates/memory`

```
crates/memory/src/
├── lib.rs
├── store.rs       # SQLite FTS — write, search
└── injector.rs    # inject relevant memories into system prompt
```

**Schema:**

```sql
CREATE VIRTUAL TABLE memory_fts USING fts5(
    content,
    source,        -- 'user' | 'agent' | 'tool'
    created_at UNINDEXED
);
```

**Flow:**

1. After each turn, extract notable facts (via a lightweight LLM call or regex)
2. Store them in `memory_fts`
3. At the start of each turn, search for relevant memories
4. Inject top-N results into the system prompt as context

**Delivers:** agent knows your name, preferences, past projects across sessions.
